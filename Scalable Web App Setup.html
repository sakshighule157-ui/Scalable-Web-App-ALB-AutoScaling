Step 11: Test Auto Scaling in Detail

After creating your Application Load Balancer (ALB) and Auto Scaling Group (ASG), the next step is to verify that Auto Scaling works correctly.

Objective

Generate high CPU usage on an EC2 instance so that the Auto Scaling Group automatically launches a new EC2 instance.

Step 11.1: Verify Auto Scaling Group

  1. Open the AWS Console.

  2. Go to EC2 → Auto Scaling Groups.

  3. Select your Auto Scaling Group (for example, Web-ASG).

  4. Check the following settings:

SettingExample Value
Desired Capacity2
Minimum Capacity2
Maximum Capacity5

You should already have 2 running EC2 instances.

Step 11.2: Connect to One EC2 Instance

Connect using SSH:

Bash
ssh -i mykey.pem ubuntu@<Public-IP>

You are now logged into the server.

Step 11.3: Install the Stress Tool

On Ubuntu, run:

Bash
sudo apt update
sudo apt install stress -y

Verify installation:

Bash
stress --version

Step 11.4: Generate High CPU Load

Run:

Bash
stress --cpu 4 --timeout 300

Explanation

  • --cpu 4 → Uses 4 CPU workers.

  • --timeout 300 → Runs for 300 seconds (5 minutes).

This increases CPU utilization significantly.

Step 11.5: Monitor CPU Utilization

  1. Go to AWS Console → EC2 → Instances.

  2. Select the instance.

  3. Open the Monitoring tab.

  4. Check the CPU Utilization graph.

The CPU usage should rise above your target threshold (for example, 50%).

Step 11.6: Watch Auto Scaling Activity

  1. Go to EC2 → Auto Scaling Groups.

  2. Select your ASG.

  3. Open the Activity tab.

You should see messages similar to:

Launching a new EC2 instance.

and later:

Successfully launched EC2 instance i-xxxxxxxx.

This confirms that the scaling policy has triggered.

Step 11.7: Verify the New Instance

Go to EC2 → Instances.

Initially:

InstanceStatus
Web-1Running
Web-2Running

After scaling:

InstanceStatus
Web-1Running
Web-2Running
Web-3Running

The third instance was created automatically by the Auto Scaling Group.

Step 11.8: Verify the Load Balancer

  1. Go to EC2 → Load Balancers.

  2. Copy the ALB DNS name.

  3. Open it in your browser:

http://your-alb-dns-name

Refresh the page several times.

If your web page contains:

HTML
<h1>Hello from $(hostname)</h1>

you will notice different hostnames, showing that the ALB is distributing requests across multiple EC2 instances.

Step 11.9: Test Scale-In

After the stress command finishes (or press Ctrl + C to stop it), CPU utilization will decrease.

After a few minutes, Auto Scaling evaluates the lower load and may terminate extra instances (while respecting the minimum capacity).

In the Activity tab, you should see a message such as:

Terminating EC2 instance because average CPU utilization is below the target value.

Step 11.10: Confirm Final State

If your ASG configuration is:

MinimumDesiredMaximum
225

then after the CPU load drops, the group should return to 2 running instances.

Expected Architecture During High Traffic

                 Users
                   │
                   ▼
      ┌──────────────────────────┐
      │ Application Load Balancer│
      └─────────────┬────────────┘
                    │
      ┌─────────────┼─────────────┐
      ▼             ▼             ▼
   EC2-1         EC2-2         EC2-3
                    ▲
                    │
      Auto Scaling launched EC2-3
      because CPU utilization exceeded
      the target threshold.

This demonstrates that your application can automatically handle increased traffic by launching new EC2 instances and reduce costs by terminating extra instances when demand decreases.
